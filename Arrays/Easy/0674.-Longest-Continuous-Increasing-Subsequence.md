# 674. Longest Continuous Increasing Subsequence

> 🟢 **Easy** &nbsp;&nbsp; 🏷️ **Arrays** &nbsp;&nbsp; 🏷️ **Sliding Window** &nbsp;&nbsp; 🏷️ **Greedy**

---

## 📝 Problem Description

Given an unsorted array of integers `nums`, return *the length of the longest continuous increasing subsequence (i.e., subarray)*. The subsequence must be **strictly increasing**.

A **continuous increasing subsequence** is defined by two indices `l` and `r` (`l < r`) such that it is `[nums[l], nums[l + 1], ..., nums[r - 1], nums[r]]` and for each `l <= i < r`, `nums[i] < nums[i + 1]`.

### Examples

**Example 1:**
* **Input:** `nums = [1,3,5,4,7]`
* **Output:** `3`
* **Explanation:** The longest continuous increasing subsequence is `[1,3,5]` with length `3`. Even though `[1,3,5,7]` is an increasing subsequence, it is not continuous as elements `5` and `7` are separated by element `4`.

**Example 2:**
* **Input:** `nums = [2,2,2,2,2]`
* **Output:** `1`
* **Explanation:** The longest continuous increasing subsequence is `[2]` with length `1`. Note that it must be strictly increasing.

### Constraints
* $1 \le \text{nums.length} \le 10^4$
* $-10^9 \le \text{nums}[i] \le 10^9$

## 💡 Core Standard DSA Patterns

### 1. Sliding Window / Dynamic Anchor Pattern
* **Concept:** Maintaining a contiguous window bounds reference using a single starting anchor pointer during sequential iteration.
* **Mechanism:** Instead of re-scanning subsequences from scratch when a condition fails, the left boundary pointer (`anchor`) updates instantly to the current index `i`. The algorithm tracks valid sequences across linear indices dynamically.

### 2. Linear State Reset Pattern
* **Concept:** Accumulating a continuous structural pattern count while executing an immediate reset whenever structural invariant criteria are violated.
* **Mechanism:** A running element counter grows incrementally by checking adjacent pairs (`nums[i] > nums[i-1]`). The scalar state completely drops back to baseline (`1`) when values flatten out or drop, avoiding memory-expensive array transformations.

## 💻 Standard DSA Implementations

### 1. Sliding Window Anchor Approach ($O(N)$ Time & $O(1)$ Space)
The standard window tracking architecture. It maintains a structural anchor coordinate to calculate subarray window spaces dynamically via current position pointers.

```java
public class Solution {
    public int findLengthOfLCIS(int[] nums) {
        if (nums == null || nums.length == 0) {
            return 0;
        }

        int maxLen = 0;
        int anchor = 0;

        // Sweep through every element index in the array
        for (int i = 0; i < nums.length; i++) {
            // Anchor Reset Guard: If elements stop strictly increasing, shift anchor to current
            if (i > 0 && nums[i - 1] >= nums[i]) {
                anchor = i;
            }
            
            // Calculate active window size dynamically and update global maximum
            maxLen = Math.max(maxLen, i - anchor + 1);
        }

        return maxLen;
    }
}
```

### 2. Scalar Iterative Counter ($O(N)$ Time & $O(1)$ Space)
An elegant, simplified variant for maximum optimization. This skips index offsets completely by tracking state lengths via a direct scalar accumulator variable.

```java
public class Solution {
    public int findLengthOfLCIS(int[] nums) {
        if (nums == null || nums.length == 0) {
            return 0;
        }

        int maxLen = 1;
        int currentLen = 1;

        // Start processing adjacent element pairs from index 1
        for (int i = 1; i < nums.length; i++) {
            if (nums[i] > nums[i - 1]) {
                currentLen++;
                maxLen = Math.max(maxLen, currentLen);
            } else {
                // Instantly drop back to baseline length of 1 if rule is broken
                currentLen = 1;
            }
        }

        return maxLen;
    }
}
```

### Detailed Evaluation Breakdown

#### 1. Zero-Allocation Traversal Efficiency
* **No GC Footprint:** Both approaches rely strictly on primitive local integer variables (`maxLen`, `anchor`, `currentLen`). They bypass object instantiation entirely, removing heap allocation stress and bypassing Java garbage collection tracking passes.
* **Cache Line Alignment:** The elements of the primitive array `int[]` sit contiguously in hardware memory. Because the loops scan the array step-by-step from index `0` to `n-1`, modern CPU systems load consecutive items directly into high-speed L1/L2 caches ahead of time, ensuring lightning-fast execution.

#### 2. The Power of Single-Pass Elimination
For continuous structures, maintaining a global table across iterations is redundant. A naive nested loops solution searches from every index, wasting time with an $\mathcal{O}(N^2)$ footprint. By applying a greedy single-pass model, whenever an item drops below or matches its predecessor (`nums[i-1] >= nums[i]`), any possible increasing subsequence starting before that index is broken. This allows the pointer to jump directly to the current item, safely dropping execution time down to a clean, stable $\mathcal{O}(N)$.


## 💡 Core Standard DSA Patterns

### 1. Functional Stream Range Reduction
* **Concept:** Transforming a structural array traversal into a functional range projection pipeline using Java 8 functional abstractions.
* **Mechanism:** Instead of updating loop variables manually, `IntStream.range()` projects intermediate states over primitive indices. State continuity calculations are combined across processing elements within stateless transformations.

### 2. State-Accumulating Reduce / Scan Pattern
* **Concept:** Maintaining intermediate track histories over consecutive element pairs within custom aggregate reduction objects.
* **Mechanism:** Since native Java 8 Streams do not natively pass state between adjacent pipeline calculations, an index projection map evaluates boundary breaks (`nums[i] > nums[i-1]`). A reduction collector keeps tabs on ongoing active lengths to continuously capture peak window sizes.

## 💻 Standard DSA Implementations

### 1. Functional Stream Reduction ($O(N)$ Time & $O(1)$ Space)
A pure Java 8 Streams framework solution. It maps array index ranges into an aggregated state object wrapper that retains runtime metadata without modifying the external source array.

```java
import java.util.stream.IntStream;

public class Solution {
    // Helper tracker class to hold state during stream reduction
    private static class StreamState {
        int maxLen = 1;
        int currentLen = 1;
    }

    public int findLengthOfLCIS(int[] nums) {
        if (nums == null || nums.length == 0) {
            return 0;
        }

        // Initialize state engine and reduce across indices sequentially
        StreamState result = IntStream.range(1, nums.length)
            .collect(
                StreamState::new,
                (state, i) -> {
                    if (nums[i] > nums[i - 1]) {
                        state.currentLen++;
                        state.maxLen = Math.max(state.maxLen, state.currentLen);
                    } else {
                        state.currentLen = 1;
                    }
                },
                (state1, state2) -> {
                    // Parallel combiner fallback logic
                    state1.maxLen = Math.max(state1.maxLen, state2.maxLen);
                }
            );

        return result.maxLen;
    }
}
```

### 2. Stream In-Place Peak Tracking ($O(N)$ Time & $O(1)$ Space)
An implementation using Java 8 Stream terminal reduction primitives to directly map raw array values against index changes.

```java
import java.util.stream.IntStream;

public class Solution {
    public int findLengthOfLCIS(int[] nums) {
        if (nums == null || nums.length == 0) {
            return 0;
        }

        // Use a container array to maintain transient state values across stream operations
        // track[0] stores current window length, track[1] stores global max length
        final int[] track = {1, 1};

        IntStream.range(1, nums.length).forEach(i -> {
            if (nums[i] > nums[i - 1]) {
                track[0]++;
                if (track[0] > track[1]) {
                    track[1] = track[0];
                }
            } else {
                track[0] = 1;
            }
        });

        return track[1];
    }
}
```

### Detailed Evaluation Breakdown

#### 1. Functional Overhead Considerations
* **Lambda Closure Costs:** Java 8 functional structures introduce tiny allocation layers due to lambda bindings and dynamic function dispatch tables. While auxiliary memory stays technically bounded at $\mathcal{O}(1)$, these overheads make execution slightly slower than pure, direct primitive loops.
* **Primitive Optimization:** By using `IntStream` instead of a generic `Stream<Integer>`, the implementation runs without performance-sapping auto-boxing or unboxing operations. This keeps memory access predictable across large input sizes.

#### 2. Sequential vs. Parallel Traversal Limitations
Streams provide clean data pipeline separation but are naturally optimized for independent element mapping. Because continuous sequences require evaluating strict dependencies between adjacent items (`nums[i]` and `nums[i-1]`), running this operation across multiple threads in parallel can break state tracking. For continuous sequence tracking, running the stream sequentially provides correct results while avoiding context-switching thread overhead.
